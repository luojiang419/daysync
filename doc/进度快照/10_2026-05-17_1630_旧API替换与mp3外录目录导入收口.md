## 本次问题定位

1. 这不是前端纯误报。
2. 实测发现当前 `127.0.0.1:17831` 上跑着的是更早一版便携包的后台：
   - `ffmpeg.root_path` 指向 `DaySync_Windows_Portable_20260517_154358`
3. 新前端连到了旧后端，造成：
   - 健康检查显示“已连接”
   - 但真正的导入请求打到旧 API 时，仍会因为旧逻辑不支持目录而报错
4. 同时还确认到第二个素材侧问题：
   - 用户给的外录音频目录全是 `.mp3`
   - 之前后端只支持 `.mov / .mp4 / .wav / .m4a`
   - 所以即便目录导入逻辑正确，音频目录也会被整体忽略

## 本次完成内容

1. 修桌面端“旧 API 占端口”问题：
   - Tauri `ensure_dev_api()` 现在会先检查当前 `17831` 上的 API 是否属于当前包自己的运行时
   - 若发现是旧便携包后台：
     - 先尝试调用 `/api/admin/shutdown`
     - 如果旧版没有这个接口，则在 Windows 上按端口直接结束旧进程
     - 再拉起当前包自己的 API
2. 后端新增本地关闭接口：
   - `/api/admin/shutdown`
   - 仅用于桌面壳在本机替换旧 API 进程
3. 媒体导入扩展支持 `.mp3`
   - 后端目录递归扫描现支持：
     - `.mov`
     - `.mp4`
     - `.wav`
     - `.m4a`
     - `.mp3`
   - 前端媒体页说明同步补上 `mp3`
4. 补测试：
   - Rust 单测覆盖运行时匹配判断
   - Python 媒体导入目录测试改为验证 `.mp3`
