## 关键根因结论

1. 旧问题不是 worker 没启动：
   - 实测 `DaySync.exe` 会成功拉起 `python -m daysync_core.bridge.worker`
2. 真正故障点在首条健康检查响应解码：
   - worker 响应 JSON 中包含中文工作区路径（当前仓库路径就含中文）
   - Windows pipe 下 Python 默认输出编码不是 UTF-8
   - Rust 侧按 UTF-8 读 `stdout`，于是首条响应在解码阶段失败并被前端判定为运行时不可用
